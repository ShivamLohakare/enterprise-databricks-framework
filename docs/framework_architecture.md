1. Business Problem-
    The organization receives data from multiple source systems and needs a scalable, reusable, and configurable framework to onboard datasets into a Databricks Lakehouse architecture.
    Traditional ETL pipelines require code changes for every new dataset, resulting in higher maintenance effort, inconsistent implementations, and slower onboarding.
    The objective is to create a metadata-driven framework that allows onboarding of new datasets through configuration changes instead of code modifications.

2. Business Requirement-
    1. Support onboarding of multiple datasets.
    2. Support CSV, Parquet, and Delta formats.
    3. Follow Bronze-Silver-Gold architecture.
    4. Capture operational audit information.
    5. Support schema evolution monitoring.
    6. Provide centralized error handling.
    7. Support metadata-driven processing.
    8. Minimize code changes during onboarding.
    9. Generate operational alerts.
    10. Support future scalability for hundreds of datasets.

5. Metadata Framework-
    i. Store dataset onboarding configurations.
    ii. Store source system information.
    iii. Store target catalog/schema/table information.
    iv. Store write strategies.
    v. Store primary key definitions.
    vi. Store schema evolution settings.
    vii. Enable onboarding of new datasets without code changes.

6. Reader Framework-
    i. Read dataset configuration from metadata layer.
    ii. Validate source file format and source path.
    iii. Apply source-specific read options.
    iv. Read source data into Spark DataFrame.
    v. Handle source read exceptions.
    vi. Log operational events.
    vii. Return DataFrame to downstream framework components.

7. Writer Framework-
   i. Read write modefrom metadata
   ii. Construct catalog.schema.table
   iii. Create target table automatically
   iv. Handle error and exception properly
   v. Logging the error in the audit log
   vi. Support overwrite (V1)
   vii. Future support for apend and merge
   viii. Create target schema/table if not present.
   ix. Support metadata-driven write strategies.

8. Audit Framework-
    i. Generate unique batch identifier.
    ii. Capture execution start event.
    iii. Capture execution completion event.
    iv. Capture execution failure event.
    v. Store processing metrics.
    vi. Store exception details.
    vii. Store execution duration.
    viii. Generate operational alerts.

9. Validation Framework-
    i. For Bronze Layer Validation, Validate source accessibility and schema drift detection.
    ii. For Silver Layer Validation, check deduplication, null, data quality, standardizaion, business rules.
    iii. For Gold Layer Validation, check business rules.

10. Schema Evolution Strategy-
     If schema drift occurs:
        i. Load data into Bronze layer.
        ii. Generate schema drift alert.
        iii. Update audit logs.
        iv. Stop Silver processing.
        v. Stop Gold processing.
        vi. Await schema review and approval. 

11. Error Handling Strategy
### Objective - The framework implements a centralized error handling mechanism to ensure consistent failure management, operational visibility, and simplified troubleshooting across all datasets and processing layers.

### Design Approach - The framework follows a fail-fast approach. Whenever an exception occurs during data ingestion, validation, or data writing, processing is immediately stopped to prevent invalid data from propagating to downstream layers. All exceptions are captured through a common error handling component which records execution details, updates audit information, and generates alerts for support teams.

### Error Handling Flow -
1. Capture the exception generated by the framework component.
2. Record failure details in the audit log table.
3. Store the exception message and execution context.
4. Update the batch status as FAILED.
5. Generate operational alerts.
6. Stop downstream processing.
7. Preserve execution metadata for troubleshooting.

### Reader Framework Error Handling
The Reader Framework handles source-level failures such as:
* File not found
* Invalid source path
* Unsupported file format
* Corrupted source files
* Permission issues
If any of these failures occur, the framework logs the error, updates the audit table, generates an alert, and terminates processing for the affected dataset.

### Validation Framework Error Handling
The Validation Framework handles:
* Schema drift detection
* Duplicate record validation
* Null value validation
* Data quality failures
* Business rule violations
Validation failures are captured in the audit framework and prevent downstream Silver and Gold processing.

### Writer Framework Error Handling
The Writer Framework handles:
* Target table access failures
* Schema mismatch issues
* Storage failures
* Write operation failures
When a write failure occurs, the framework records the error details, updates the audit status, and stops further processing.

### Schema Evolution Error Handling
If schema drift is detected, the framework allows data ingestion into the Bronze layer to preserve source data. An alert is generated and the event is recorded in the audit logs.
Silver and Gold processing are halted until the schema change is reviewed and approved.

### Audit Integration
Every failure event is recorded with:
* Batch ID
* Dataset Name
* Source Path
* Target Table
* Start Time
* End Time
* Processing Duration
* Status
* Error Message
This enables complete traceability and simplifies operational support.
